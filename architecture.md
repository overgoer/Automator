# Архитектура проекта Automator

## Цель
ИИ-контент креатор: анализ постов Telegram канала → генерация контента через ИИ → отправка готового поста (email/Telegram)

## Контекст проекта
- **Канал**: @eddytester (тестирование QA)
- **Стиль**: технический, без воды, с эмодзи 🤘
- **Расписание**: каждые 2 дня
- **Хостинг**: GitHub Actions

---

## 1. Выбор технологий

### Бэкенд
- **Язык**: Python 3.11+
- **Причина**: богатые библиотеки для Telegram, ИИ, планировщиков

### Хранение данных
- **SQLite** — для начала (простота, не требует отдельного сервиса)
- **Структура**:
  - `posts` — скачанные посты (id, text, date, processed)
  - `generated` — сгенерированный контент (id, post_id, content, model, date)
  - `config` — настройки (последний обработанный пост, промпты)

### Telegram
- **Библиотека**: `telethon` (async, надежная)
- **Доступ**: только чтение канала (не нужен бот в канале)
- **Что скачиваем**: текст постов, дата, id

### ИИ-модель
- **Провайдер**: DeepSeek API или Qwen API (через OpenAI-compatible endpoint)
- **Причина**: стабильный HTTP API, недорогой, хороший контекст
- **Резерв**: второй провайдер на случай недоступности

### Уведомления
- **Email**: SMTP (через `aiosmtplib`)
- **Telegram Bot**: `aiogram` для отправки сообщений

### Планировщик
- **APScheduler** — cron-подобное расписание, перезапуск после сбоев

## 4. Хостинг

### Вариант A: GitHub Actions (рекомендуется для начала)
- **Плюсы**: бесплатно, не нужно админить сервер
- **Минусы**: запуск по расписанию (не чаще раза в 5 мин), лимиты
- **Реализация**: workflow с cron-триггером

### Вариант B: Ваш сервер (для продакшена)
- **Docker Compose** — контейнеризация
- **Systemd** — автоперезапуск при падении
- **Плюсы**: полный контроль, любое расписание
- **Минусы**: нужно админить

### Решение
Начинаем с **GitHub Actions** для простоты. При необходимости масштабирования — переход на сервер.

## 5. Надежность (Toyota Land Cruiser принцип)

### Обработка ошибок
- Retry с экспоненциальной задержкой для API вызовов
- Логирование всех ошибок в файл
- Уведомление об критических сбоях

### Восстановление
- Сохранение позиции последнего обработанного поста
- Идемпотентность — повторный запуск не дублирует контент

### Мониторинг
- Логирование каждого этапа
- Health-check endpoint (опционально)
